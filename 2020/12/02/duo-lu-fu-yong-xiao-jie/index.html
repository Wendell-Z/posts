多路复用小结

基础—Linux系统架构

Linux中的架构分层

Due to the complexity of developing and maintaining the kernel(内核), only the most essential and performance-critical code are placed in the kernel. Other things, such as GUI, management and control code, typically are programmed as user-space applications. This practice of splitting the implementation of certain features between kernel and user space is quite common in Linux.

用户态 CPU的一种状态

应用程序运行的空间，为了让应用程序能够访问到内核管理的资源:CPU&内存等，提供了接口访问，就是系统调用层（为了保护计算机硬件资源）。

内核态 CPU的一种状态

一种特殊的程序，控制计算机资源：CPU资源，分配内存资源，提供稳定环境运行。

应用程序通过操作系统，从而将CPU转为内核态。可能有多种CPU状态，但至少有这两种。

Unix IO模型

一个IO操作从用户态转为内核态包括两个阶段

  • 等待数据
  • 从内核进程复制数据

对于Socket IO,就是等待网络数据到达，从网卡到到内核缓冲区，内核缓冲区到应用进程缓冲区

Blocking IO

阻塞式IO

  • 应用发起recvFrom系统调用，应用被阻塞

  • 内核等待数据准备好

  • 数据准备好，将数据复制到应用进程缓冲区

  • 应用进程拿到数据进行处理

缺点：

  • 创建线程，底层调用clone()函数，线程成本高
  • 耗费资源，要为线程开辟栈空间
  • 线程切换浪费资源，保留上下文

Non-Blocking IO

非阻塞式IO

  • 应用发下recvFrom系统调用，应用不会被阻塞，有数据就有，没有就没有
  • 应用轮询内核（recvFrom），数据准备好就阻塞取数据

IO多路复用

  • 通过select或者poll，发起系统调用，
  • select中有多个socket，有一个准备好的时候
  • 再次发起系统调用将数据拷贝到应用进程缓存

只有一个客户端，不如阻塞IO，但处理多各客户端就需要多路复用提高CPU使用效率

常用方法是将非阻塞IO和多路复用结合起来

信号驱动

  1. 应用进程发起一个信号，告诉内核要什么文件，然后立马返回
  2. 内核准备好数据
  3. 应用进程收到信号发起recvfrom系统调用来阻塞取数据

AIO

异步非阻塞IO

  1. 应用进程发起aio_read系统调用，告诉内核要什么文件，然后立马返回
  2. 内核准备好数据, copy到应用进程, 然后给应用进程发aio_read指定的信号
  3. 应用进程收到信号，开始处理数据,整个过程应用进程都没有被阻塞

多路复用

I/O多路复用通过使用一个系统函数，如select, 可以同时等待多个fd的可读，可写等状态。

Select发布以后, I/O就能复用了，你可以询问内核哪些fd准备就绪了，然后去发系统调用读数据,
读fd的过程是阻塞的，使用select可以避免无意义的阻塞, 这样即使只有一个进程也可以处理多个socket fd的读写

Non-blockingAPI，但是它和select不同，select是帮助你多路复用, Non-blockingAPI是指你在读fd的时候不会阻塞等待数据准备和内核拷贝数据的完成, 这个时候进程可以干别的事情

Non-Blocking也可以实现类似select的功能，不过那是应用层去做select的功能，也就是应用层需要轮询描述符是都就绪， 假如没有就绪，对fd的read会直接返回一个error, 代表没有数据可读等。
轮询的缺点是浪费了太多CPU时间，因为read是要发系统调用的，进程会从应用态切换到内核态, 每次这个切换过程都是资源的一种浪费.

//放入N个文件描述符合 监听读写状态

select(nfds read_set) bitmap

Select函数

// 用户态这边有个数组fds[]存的是socket,fds[i]会和set对应上
// 文件最大描述符（maxfdpl字段的意义：作为bitmap的limit 读的时候只读0-limit ） 读文件描述符 写文件描述符 异常文件描述符（IO状态）
// set是一个bitmap 大小时1024 最大支持1024个文件描述符
// 函数运行时会将应用程序中的文件中描述符放入拷贝到内核态 内核态直接判断哪一个数据来了
// 如果不拷贝，那么每次在应用程序中判断哪个数据来，都要进行用户态和内核态的切换
// 函数阻塞 直到有数据来 返回值表示fd_set里就绪的元素总个数,包括读,写,异常fd_set
// 有数据时，哪个fd有数据就将set对应的位置置位为1，表示有数据 同时将数据拷贝到用户态 函数返回
int select(int maxfdpl, fd_set *readset, fd_set *writeset,
           fd_set *exceptset, const struct timeval *timeout);
//select不再阻塞返回后，就遍历返回值 有数据的就去匹配对应的socket读出处理 另将set置位
//优点：将监听的fd集合放到内核态，让内核帮忙监听，节省了应用程序询问IO是否到来的系统切换
//缺点：
//1、默认大小1024(32位)，大小不够
//2、fdset不可重用 每次要重新置位 
//3、仍需要用户态到内核态的切换（select返回时） ，并且内核态内部要遍历O(N)
//4、应用程序遍历是O(N) N越大时间越长，没数据的读写就是无效的系统调用

poll函数

//比bitmap好
struct pollfd {
  int fd; // 需要检查的fd 文件描述符
  short events; // 该fd感兴趣的事件 pollin pollout
  short revents; // 该fd当前发生的事件 
}
//将socket也就是fd放到pollfd数组中 并赋值events中在意的事件 revents置位初始默认值
//函数阻塞调用 copy到内核态监听 有一个来数据就设置revents copy到用户态返回
//pollfd的数组，不止1024   
int poll(struct pollfd *fdarray, unsigned long nfds, int timeout);
//函数返回 遍历返回数组 匹配fd取数据操作，并将revents置为默认值
//优点：
//1.将fd放到内核态监听，节省切换
//2.数组存放fd，没有了1024的限制
//3.event支持更多地事件类型
//4.revents可以复用
//缺点：
//1.返回时内核态到用户态切换 
//2.O(N)遍历 无效的系统调用

epoll

//epoll_create相当于在内核态开启一片空间，用于存放监听的fd
//epfd就是这片区域的位置 ep的文件描述符
int epfd = epoll_create(int size);
//每来一个新的fd，就调用epoll_ctl，将它放入epfd，进行监听
//创建完epoll_file之后，接着是使用epoll_ctl添加需要关注的文件，epoll_ctl支持三种操作：
//插入事件
//删除事件
//修改事件
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
//通过epoll_wait阻塞等待fd有数据
//返回的内容是有事件的fd，没事件的不会返回
int epoll_wait(int epfd, struct epoll_event * events, int maxevents, int timeout);
//优点:
//1.fd放入内核态监听
//2.只返回有数据的fd，不再用遍历，不再用整个fd数组切换
//3.通过mmap加速了内核态到用户态之间的切换
//4.支持水平触发(LT，边缘触发(ET)
//如果是边缘触发模式，下次如果再使用epoll_wait，进来这里的时候，如果这个文件的事件没有产生，那就不会读到这个文件，不会去判断有没有事件发生。
//如果你第一次调用epoll_wait且返回了这个文件的事件，但是你没有去处理，那下一次调用epoll_wait的时候，如果期间没有事件产生，epoll_wait将不会返回这个文件，因为已经被从队列里面删了，所以用这个模式的时候你必须确保每一次的epoll_wait返回列表里面的事件你都已经处理掉，否则会漏掉。
//如果使用的是水平触发模式，那就继续把epitem插入队列，下一次epoll_wait的时候，如果事件还存在，则会继续返回，直到你处理完。
//所谓队列：是epoll维护的一个就绪事件的队列
